# 产品洞察笔记

> 从「和弦灵感生成器」项目中萃取的产品思维，适用于后续项目。

## 1. 找到真正的用户：专业和纯小白之间

**核心问题：不是所有人都是你的用户。** 初版 PRD 的目标用户写了「卧室唱作人」「即兴爱好者」「乐理学习者」三张画像，看起来很完整，但实际上没有回答最关键的问题：谁才真正需要这个产品？

- **专业创作者** → 常见和弦走向烂熟于心，不需要工具提醒。他们需要的是更快的 DAW 快捷键，不是随机和弦。
- **完全不懂乐理的小白** → 连「和弦走向」是什么都不知道，打开页面看到「I-V-vi-IV」只会一头雾水。
- **中间层——我们的用户** → 学过一点乐理，知道走向的概念，听过 1-6-4-5，但对更丰富的走向没有积累。拿着一把吉他脑子一片空白，想写歌但只会书里那两套进行。**知道「我需要更多素材」但不知道去哪找。**

**原则：** 写 PRD 时不要只列「谁会用到」，要明确「谁一定不会用到」。排除专业用户和小白用户后，产品决策反而更清晰——不做乐理教学，不做进阶功能，只做「给你素材」。

## 2. JTBD：翻译用户说的话

**用户描述的是「他们想象中的解决方案」，不是「他们真正的问题」。**

- 用户说「我要更多和弦」→ 翻译：我手上这几个已经用腻了，弹出来的东西都一个味
- 用户说「能播放一下吗」→ 翻译：我看到和弦符号不确定它听起来对不对，我害怕弹出难听的东西
- 用户说「这个风格不够多」→ 翻译：连续生成结果之间差异太小，我需要新鲜感

**实操方法：**
1. **追问法**：用户提一个功能需求，追问「你上一次遇到这个问题是什么时候？当时发生了什么？」逼他从抽象需求切换到具体场景
2. **替代方案法**：问他「现在没这个工具的时候怎么解决？」看到真实行为
3. **动机层次法**：把功能需求往上一层抽象——「播放功能」→「我想知道好不好听」→「我害怕弹错」→ 他要的是安全感

## 3. 功能选择：不是「做什么」，是「证明什么」

**MVP 不是砍出来的一半功能，是最少的东西验证最关键的假设。**

本项目的核心假设只有一个：**用户看到一组随机的和弦进行，会不会立刻拿起乐器弹？** 验证这个只需要三样东西：调性选择器、风格选择器、随机生成按钮。其他全部砍掉。

- 收藏？→ 没验证核心价值之前，收藏的数据毫无意义
- 指位图？→ 用户会弹和弦，不需要你教他怎么弹
- 音频播放？→ 用户有乐器，自己弹。他不是要你帮他弹，是要你告诉他弹什么

**优先级矩阵（2×2）：**
- 高影响 × 低 effort = 立即做（随机生成、多风格）
- 高影响 × 高 effort = 分批做（收藏）
- 低影响 × 低 effort = 闲时做
- 低影响 × 高 effort = 不做（音频播放、指位图、AI 生成）

把这个矩阵放在每次产品决策的最前面。

## 4. 用户故事：so that 要能反推产品决策

**bad so that：** 「As a 用户, I want 和弦生成功能, so that 我能生成和弦」——动机等于没说，开发看到只能猜。

**good so that：** 「so that 我能在 3 步内从空白到弹琴，注意力始终留在乐器上」——这可以直接反推：不能加 loading 动画、不能要求登录、不能有复杂设置。

**验收标准必须是可测量的：** 「Then 和弦生成成功」没法判断。「Then 操作步数 ≤3 步，全程无需滚动阅读说明」——可验证。

## 5. PRD 版本管理：增量写，不重复写

**问题：** 初版有两个 PRD（v1.0 和 v1.0.1），各自完整但结构不同，v1.0.1 重复了大量 v1.0 已有内容。

**教训：**
- v1.0 是完整的基线 PRD，7 章模板从头写到尾
- v1.0.1 用同样的模板但只写变更：每个章节标注「相对于 v1.0 的变化」，无变化章节直接写「保持不变，参见 v1.0」
- 版本之间通过文档信息里的「上一版本 / 下一版本」链和附录链接串联

**这样做的收益：**
- v1.0.1 从 127 行压到 ~60 行
- 读完 v1.0 的人不需要重读一遍背景和动机
- 版本演进链清晰，变更可追溯

## 6. 写好一个 PRD 的成本很低，写坏了的成本很高

**写烂了的代价：** 我自己在写 v1.0 PRD 时对目标用户、产品边界都不清晰，导致后面不断追加、修修补补（v1.0.1、v1.0.2…），每次补充都在已有的模糊框架上打补丁，PRD 逐渐失控。

**花了 1 小时整理清楚的收益：** 用 brainstorming 流程重写后——
- 目标用户定义从三张模糊画像变成一句话：「略懂乐理但卡在套路的中间层爱好者」
- 功能范围的每个「不做」都有了明确理由（不是觉得不重要，是先不验证）
- 版本规划不再拍脑袋，每个后续版本都有触发条件

**原则：花 1 小时把 PRD 写清楚，比花 10 小时在模糊的方向上开发划算。**
