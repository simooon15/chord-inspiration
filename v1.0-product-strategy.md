# 第三步：产品策略 —— 和弦灵感生成器

> 基于 JTBD 用户洞察 + 用户故事，定义 MVP 范围、优先级、风险与下一步。

---

## 1. MVP 范围定义

### MVP 一句话

> 一个单页网页，顶部有调性下拉框（默认 C 调），用户选风格点一下按钮，立刻看到一组随机和弦进行（含纳什维尔数字级数标注）。不需要登录、不需要设置、打开就能用。

### MVP In / Out

| 维度 | MVP 做 | MVP 不做 |
|---|---|---|
| 风格数量 | 5 种（流行/民谣/摇滚/爵士/R&B） | 自定义风格、小众风格 |
| 和弦显示 | 和弦名称 + 构成音 + 调性选择 + 级数标注（纳什维尔数字，如 C 调 1-5-6m-4: C-G-Am-F） | 五线谱、吉他指位图、音频播放 |
| 交互 | 顶部调性下拉框（默认选中并标注「C 调」，切换即时生效）→ 选风格 → 点生成 → 看结果（含纳什维尔数字级数 + 和弦名称）；一键换一组 | 历史记录、撤销、分享 |
| 保存 | 不做（P3 推迟） | 收藏、导出、账号系统 |
| 设备 | 仅移动端竖屏（手机优先） | iPad（V2 响应式）、桌面（暂不做） |
| 技术 | 纯前端 HTML/CSS/JS，无后端 | 数据库、API、用户系统 |

### MVP 切在这里的理由

P0+P1 解决「不知道弹什么 + 总在套路里循环」两个核心痛点，这两个痛点频率最高、痛苦最强烈。P2（极简 UI）不是独立功能，而是所有功能的公共约束——每个功能都要满足 ≤3 步。P3（收藏）涉及数据持久化，引入技术复杂度但解决的是「遗忘」问题——用户还没用上，先不用管「忘了怎么办」。

**本次新增：调性选择 + 和弦级数（纳入 MVP）。** 用户决策：采用 Option C——页面顶部固定调性下拉框，默认 C 调并明确标注，切换即时生效无需重新生成。级数体系首版用纳什维尔数字（1/5/6m），后续版本加切换按钮支持罗马数字（I/V/vi）。选择纳什维尔首发的理由：核心用户是流行/民谣唱作人，纳什维尔数字是他们的通用语言。纳入 MVP 理由：技术成本极低（一个 `<select>` + 音名映射函数，约 5 行 JS），但对用户价值高——没级数就看不懂和弦之间的相对关系，没法转调。

## 2. 功能优先级矩阵

按 Impact（用户价值）× Effort（实现难度）二维评估：

```
高影响 │
       │  P0 随机生成          P1 多风格
       │  (立即做)             (立即做)
       │
       │  P2 极简UI约束         P3 收藏
       │  (作为设计原则)        (V2)
       │
低影响 │
       └──────────────────────────────
         低 effort              高 effort
```

| 优先级 | 功能 | 用户价值 | 实现难度 | 策略 |
|---|---|---|---|---|
| P0 | 选调 + 选风格 → 随机生成和弦 + 级数标注 | 高：直接解决「没起点」+「不能转调」 | 低：静态数据 key-value 映射 + Math.random | MVP 立即做 |
| P1 | 5 种风格，每次不重复 | 高：解决「套路循环」 | 低：每种风格准备 10+ 组数据 | MVP 立即做 |
| P2 | 3 步内完成核心操作 | 高：不解决则 P0/P1 白做 | — | 设计原则，不单独开发 |
| P3 | 收藏喜欢的进行 | 中：锦上添花 | 中：需要 localStorage | V2 |
| P3.5 | iPad 响应式适配 | 中：iPad 看谱场景真实存在 | 低：几行 CSS media query | V2（与收藏同版） |
| P4 | 和弦指位图 | 低-中：辅助弹奏 | 高：需要图片/Canvas | V3 |
| P5 | 音频播放 | 低-中：辅助确认 | 高：需要音频库 | 不做（用户有乐器，自己弹） |

### P2 作为设计原则的说明

P2「3 步内完成」单独作为一个故事可能令人困惑——它不是功能。把它定为**所有功能的验收门槛**更合理：任何新增功能，用户操作步数增量必须为 0 或 1。这是一个产品原则，不是一个开发任务。

## 3. 关键假设与风险

### 假设 1：和弦数据质量足够好，用户不觉得「Low」
- **假设**：精选 50+ 组专业和弦进行（覆盖 5 种风格），用户会觉得有用而非廉价
- **验证方式**：自己做出来先给 3 个玩乐器的朋友试，问「你会再次打开吗？」
- **如果假设错了**：补数据，加入更多风格特征的进行（爵士必须有七九和弦、流行必须有经典 1-5-6-4 变体）

### 假设 2：用户不需要听到声音就能判断和弦好不好
- **假设**：目标用户（已有乐器）看到和弦名称就能在乐器上试，不需要播放
- **验证方式**：观察朋友试用时，是否有人说「这能放一下吗？我想听」
- **如果假设错了**：V2 可以用 Web Audio API 做一个简单的钢琴/吉他音色预览

### 假设 3：随机生成带来惊喜感，而不是焦虑感
- **假设**：用户会喜欢「每次不一样」的新鲜感，不会因为「不能指定调性」而挫败
- **验证方式**：观察用户是否连续点击超过 5 次生成——如果是，说明随机性是上瘾的
- **如果假设错了**：加一个简单的「调性」过滤选项

### 风险矩阵

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| 和弦数据太水，用户感觉像 AI 瞎编 | 中 | 高——核心价值崩溃 | 全部数据人工筛选，不用 AI 生成 |
| 随机性导致重复率高 | 中 | 中——用户觉得不新鲜 | 增加数据量（每种风格 ≥15 组），加简单去重逻辑 |
| 移动端浏览器兼容差 | 低 | 中——部分用户打不开 | 只用标准 HTML/CSS，不做复杂布局 |

## 产品 Sense 解读

### 这一课：MVP 不是「做不完的项目砍掉一半」，是「最少的东西验证最关键的假设」

**核心概念。** MVP 的 M 是 Minimum，不是 Mini。Mini 是你想做的全砍一半；Minimum 是找到那个刚好能回答「这个方向对不对」的临界点。如果把产品比作实验，MVP 就是实验设计——变量要少到你能归因，样本要够到你能下结论。

**为什么重要。** MVP 切错位置的代价比你想象的大。切太少：用户不知道你在干嘛，反馈全是你已知的（「没登录很难受」——你本来就知道登录有用）。切太多：做了 3 个月上线发现没人要，浪费了大量时间。MVP 的艺术在于：做出刚刚好的东西，让用户能给「有意义的反馈」而不是「废话反馈」。

**这个项目的 MVP 逻辑。** 我们选了 P0+P1 作为 MVP，砍掉了 P3 收藏。逻辑不是「收藏不重要」，而是「用户还没验证核心价值之前，收藏的数据毫无意义」。如果用户第一次使用后不回来，收藏列表永远是空的——你做空的列表干嘛？先把「用户愿不愿意用完一次再回来」验证了，再考虑「回来之后怎么留住他上次的东西」。

**优先级矩阵怎么用。** 高影响 × 低 effort = 立即做（P0/P1）。高影响 × 高 effort = 分批做，先找低成本验证方式。低影响 × 低 effort = 闲时做。低影响 × 高 effort = 不做。这个 2×2 就是你以后做任何产品决策的默认思维框架。

**输出怎么驱动下一步。** 策略定了之后，下一步 prototype 就有明确边界了：一个手机竖屏页面，5 个风格按钮，一个生成按钮，一个结果显示区。没有设置页、没有收藏按钮、没有登录。原型不需要设计这些。
