# 第二步：用户故事 —— 和弦灵感生成器

> 基于第一步 JTBD 分析输出，按 P0→P3 优先级拆解。

---

### User Story 1 (P0): 按风格随机生成和弦进行

- **Summary:** 让创作新手一键获得符合风格的和弦灵感，打破「不知道弹什么」的僵局。

#### Use Case
- **As a** 吉他/键盘爱好者，拿着乐器想写点东西但脑子一片空白
- **I want to** 选择调性、选择风格，获得一组随机的和弦进行（同时显示和弦名称和级数）
- **so that** 我能立刻用自己熟悉的调开始弹，还能看懂和弦之间的相对关系

#### Acceptance Criteria
- **Scenario:** 用户首次使用，选调、选风格后获得一组和弦（含级数）
- **Given:** 我打开网页，页面顶部有一个调性下拉框，默认选中并标注「C 调」
- **and Given:** 无需登录、无需设置
- **When:** 我保持 C 调不变（或切换到 G 调）、选择风格（如「流行」）、点击「给我灵感」
- **Then:** 页面显示当前调性下的一组和弦进行，同时用纳什维尔数字标注级数（如「C 调 1-5-6m-4: C-G-Am-F」或「G 调 1-5-6m-4: G-D-Em-C」），每个和弦清晰可读

- **Scenario:** 切换调性，结果显示即时更新
- **Given:** 当前显示 C 调流行进行「C 调 1-5-6m-4: C-G-Am-F」
- **When:** 我将顶部的调性下拉框从 C 切换到 G
- **Then:** 同组级数自动映射为 G 调和弦「G 调 1-5-6m-4: G-D-Em-C」，无需重新点击生成

- **Scenario:** 连续生成两次，结果不同
- **Given:** 我已经生成了一次和弦
- **When:** 我不换调性和风格，再次点击「给我灵感」
- **Then:** 显示一组新的和弦进行，与上一次不完全相同

### User Story 2 (P1): 多种风格覆盖，每次有新鲜感

- **Summary:** 让用户跳出舒适区，接触到不同风格的和弦语言，避免写出千篇一律的东西。

#### Use Case
- **As a** 独立唱作人，写了几首歌都感觉一个味
- **I want to** 在不同风格之间切换（爵士/流行/民谣/摇滚/R&B），每次生成结果都不同
- **so that** 我能接触到不熟悉的进行套路，写出风格更多样的作品

#### Acceptance Criteria
- **Scenario:** 在不同风格之间切换
- **Given:** 我当前在「流行」风格
- **When:** 我切换到「爵士」并生成
- **Then:** 显示的和弦包含爵士特征进行（如 2-5-1 变体，含七和弦/九和弦），与流行风格有明显差异

- **Scenario:** 同一风格连续生成，不出现完全重复
- **Given:** 我连续生成 5 次「民谣」风格
- **When:** 我查看这 5 次结果
- **Then:** 每组和弦进行不完全相同（至少有 1 个和弦不同）

### User Story 3 (P2): 极简操作，三步内完成

- **Summary:** 整个核心流程不超过 3 步，不打断创作心流。

#### Use Case
- **As a** 正在即兴弹琴的音乐爱好者
- **I want to** 在尽可能少的操作内获得一组和弦灵感
- **so that** 我的注意力始终留在乐器上，不会被复杂的界面打断

#### Acceptance Criteria
- **Scenario:** 首次使用，从打开到看到和弦
- **Given:** 我首次打开这个网页，调性已默认选中 C 调无需操作
- **When:** 我完成从进页面到看到和弦的全过程
- **Then:** 操作步数 ≤ 3 步（打开页面 → 选风格 → 点生成），调性切换不干预时不增加步数；即使切换调性，操作步数仍 ≤ 3 步（切调 → 选风格 → 点生成）

- **Scenario:** 重复使用，一键刷新
- **Given:** 我已经生成过一次
- **When:** 我想再换一组
- **Then:** 我只需点击一次「再换一组」按钮（1 步），无需重新选风格

### User Story 4 (P3): 收藏喜欢的进行

- **Summary:** 让用户在灵感涌现时能随手保存，不怕忘了。

#### Use Case
- **As a** 经常在碎片时间摸琴的爱好者
- **I want to** 保存我喜欢的和弦进行
- **so that** 第二天打开还能找回昨天的灵感，不因为遗忘而丢掉好的进行

#### Acceptance Criteria
- **Scenario:** 收藏一组当前显示的进行
- **Given:** 页面上正显示一组我喜欢的和弦
- **When:** 我点击「收藏」按钮
- **Then:** 该和弦进行被保存，按钮状态变为「已收藏」，收藏列表中有这条记录

- **Scenario:** 查看已收藏的进行
- **Given:** 我至少收藏过 1 组和弦
- **When:** 我点击「我的收藏」
- **Then:** 显示所有已收藏的和弦进行列表，每组包含风格标签和和弦内容

- **Scenario:** 收藏列表为空时
- **Given:** 我还没有收藏过任何和弦
- **When:** 我点击「我的收藏」
- **Then:** 显示空状态提示：「还没有收藏，去生成一组喜欢的和弦吧」

### User Story 5 (V2): iPad 响应式适配

- **Summary:** 让 iPad 用户（尤其放在谱架上的）获得与手机一致的体验，屏幕更大但操作同样简洁。

#### Use Case
- **As a** 习惯把 iPad 放在谱架上边看边弹的音乐爱好者
- **I want to** 在 iPad 上打开同一个网页，和弦内容利用大屏显示得更清楚
- **so that** 我不需要弯腰看手机，也不必下载一个单独的 App

#### Acceptance Criteria
- **Scenario:** iPad 竖屏访问，布局自适应
- **Given:** 我在 iPad Safari 中打开网页（竖屏）
- **When:** 页面加载完成
- **Then:** 布局利用 iPad 屏幕宽度，和弦文字和按钮等比放大，不出现手机版硬拉伸或大片空白

- **Scenario:** iPad 横屏访问
- **Given:** 我将 iPad 横放（横屏）
- **When:** 页面内容重新排列
- **Then:** 风格按钮单行排列、结果显示区居中，不超出可视范围

- **Scenario:** 手机和 iPad 功能一致
- **Given:** 同一组和弦在手机和 iPad 上分别打开
- **Then:** 两个设备的核心功能（选调、选风格、生成、刷新）完全一致，没有因设备不同而隐藏或新增的功能

## 产品 Sense 解读

### 这一课：用户故事不是「功能描述」，是「行为 + 动机 + 验收标准」

**核心概念。** 用户故事的结构看起来很简单——As a / I want / so that ——但大部分新手 PM 写出来的是披着故事皮的功能清单。「As a 用户, I want 和弦生成功能, so that 我能生成和弦」——这什么都没说。因为它只有动作，没有人物和动机。

**为什么重要。** 如果你写不出「so that」背后真正的用户动机，说明你还没理解用户为什么需要这个东西。这种故事交给开发，开发只能猜你的意图，最后做出来的功能和你想的可能完全不一样。更关键的：没有清晰验收条件的故事，你没法判断「做完没有」。开发说做完了，你说不对，怎么判断？验收条件就是双方都认可的「完成」的定义。

**这个故事里的好坏对比。** 
- 坏：「As a 用户, I want 和弦生成, so that 我能看到和弦」——动机是废话（看到和弦就是为了看到和弦？）
- 好：「As a 吉他爱好者, I want 选风格后随机获得一组和弦, so that 我能立刻有一个起点开始弹」——动机是「打破空白僵局」，这决定了产品要快，不能有任何额外步骤。

验收条件同理：
- 坏：「Then 和弦生成成功」
- 好：「Then 操作步数 ≤ 3 步，无需滚动或阅读说明」——可测量、可验证、直接对应「不打断心流」这个 JTBD 洞察。

**输出怎么驱动下一步。** 这四个用户故事会直接成为产品策略工作坊（Step 3）的输入——我们会拿 P0~P3 的排序去检验：MVP 到底切在哪里？P2 的「3 步内完成」是独立故事，还是应该作为所有故事的共同约束？这些问题在下一步做策略时会被翻出来重新审视。
