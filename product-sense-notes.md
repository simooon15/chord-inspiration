# 产品 Sense 笔记 · 和弦灵感生成器

> 从 V1.0 项目中萃取的产品思维框架，适用于后续项目参考。

## JTBD：把「用户说的」翻译成「用户真正需要的」

**核心概念。** 用户描述的是「他们想象中的解决方案」，不是「他们真正的问题」。PM 必须追问：用户说「我要更多和弦」，翻译过来是「我手上这几个已经用腻了，弹出来的东西都一个味」；用户说「能播放一下吗」，翻译过来是「我看到和弦符号不知道它听起来对不对」。

**实操方法。**
1. **追问法**：用户每提一个功能需求，追问「你上一次遇到这个问题是什么时候？当时发生了什么？」逼他从抽象需求切换到具体场景。
2. **替代方案法**：问他「你现在没有这个工具的时候，是怎么解决的？」看到真实行为。
3. **动机层次法**：把用户说的功能往上抽象一层——「播放功能」→「我想知道这和弦听起来怎么样」→「我害怕弹出难听的东西」→ 他要的是安全感和确认感。

## 用户故事不是「功能描述」，是「行为 + 动机 + 验收标准」

**核心概念。** As a / I want / so that 看起来简单，新手 PM 写成「As a 用户, I want 和弦生成功能, so that 我能生成和弦」——动机等于没说。开发看到这种故事只能猜。好的 so that 能反推产品决策——「立刻有一个起点开始弹」→ 产品要快、不能有加载动画、不能要求登录。

**验收条件是「做完」的共同定义。** 「Then 和弦生成成功」没法判断。必须可测量：「Then 操作步数 ≤3 步，无需滚动或阅读说明」。

## MVP 不是「砍掉一半功能」，是「最少的东西验证最关键的假设」

**核心概念。** MVP 的 M 是 Minimum，不是 Mini。切太少：用户觉得「这东西还没做完」，给不出有意义的反馈。切太多：做了 3 个月上线没人要。MVP 的艺术在于做出刚刚好的东西，让用户能给「有意义的反馈」。

**本项目为例。** 选了 P0+P1（生成+多风格）作为 MVP，砍掉了收藏。逻辑不是「收藏不重要」，而是「用户还没验证核心价值之前，收藏的数据毫无意义」。先验证「用户愿不愿意用完一次再回来」，再考虑回来之后的事。

**优先级矩阵。** 高影响 × 低 effort = 立即做。高影响 × 高 effort = 分批做。低影响 × 低 effort = 闲时做。低影响 × 高 effort = 不做。这个 2×2 是任何产品决策的默认思维框架。
